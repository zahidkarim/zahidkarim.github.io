Introspective into if statements

The take away from this section is to understand Ruby’s if statement essentially evaluates true and false and then acts accordingly.

But if statements can be enablers. Here’s how:

Say we have an Book class and the book class has a find method. “` Book.find("to kill a mockingbird”) => <#Book:03939…

Book.find(“”) => nil When we call `find()` we get a `Book` object back but when we call `find()` on and empty string we get `nil`. So say we want to iterate over a collection of `Books` titles = [“To Kill a Mocking Bird”, “”, “The Gambler”] allbooks = titles.each { |t| Book.find(t) } => [<#Book:8932>, nil, <#Book:7823>] allbooks.each { |book| book.title } => “To Kill a MockingBird” => ###Error Occurs Here This gets us thinking that `nil` went from being nothing to being something. Before we were fine with nil being in our code but now we are sending something to a `Null Object` we need to consider how to deal with it. We could use some built in methods like `compact(), send(), try()` or we could use abstraction and create a `MissingBook` Class that can return an object which can support a `title` method. Both the `Book` and `Missing Boo` classes can then be called to collect all Books: titles = [“To Kill a Mocking Bird”, “”, “The Gambler”] allbooks = titles.each { |t| Book.find(t) || MissingBook(t) } => [<#Book:8932>, <#MissingBook:8432>, <#Book:7823>] allbooks.each { |book| book.title } => “To Kill a MockingBird” => “Missing Book” => “The Gambler” “ **Note:** this is a key concept here. Understandingnil` has a major role in how we develop code.

Condition Adverse | Message Centric Abstract Seeking

The next key concept Sandy Metz brings up is the thought that we should approach conditionals adversely and think of objects as things we send messages to, Message Centric.

Lets say we have a Playlist class: ”` class Playlist def songs [“Song 1”, “Song 2”, “Song 3”] end

def list
    songs.each { |song| puts "#{song} \n" }
end

end

Playlist.new.list => “Song 1” => “Song 2” => “Song 3” Now say we need a `RandomPlaylist`: Here we can use inheritance here to solve our issue class RandomPlaylist < Playlist def songs super.shuffle end end

RandomPlaylist.new.list => “Song 2” => “Song 3” => “Song 1” Now if we want to add a `DuplicatePlaylist` where all the songs appear twice we can also use inheritance: class DuplicatePlaylist < Playlist def songs duplicate = super.dup super.push(duplicate).flatten end end

DuplicatePlaylist.new.list => “Song 1” => “Song 2” => “Song 3” => “Song 1” => “Song 2” => “Song 3” Here is where things get hairy. What if we wanted a `DuplicatedRandomPlaylist`. Here is where inheritance gets dicey and we need to move to composition. class Playlist SONGS = [“Song 1”, “Song 2”, “Song 3”]

def initialize(length:, type:)
    @songs = length.set_length(SONGS)
    @formatted = type.organize(@songs)
end

def list
    @formatted.each { |song| puts "#{song} \n" }
end

end

class DoubleLength def set_length(songs) duplicate = songs.dup songs.push(duplicate).flatten end end
